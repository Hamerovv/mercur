<objective>
Restructure root `Dockerfile` (the apps/api image — `bookshook-marketplace-api`) so the slow `npm install --omit=dev` layer is cached on package.json content, not invalidated on every source change.

Purpose: `npm install --omit=dev --legacy-peer-deps` inside `.medusa/server` takes ~6 minutes. Today it sits AFTER `RUN cd apps/api && bunx medusa build`, which writes new files into `.medusa/server` on every source change, busting the install cache every build.

Output: A multi-stage `Dockerfile` where the runtime stage's prod-deps layer is keyed only on the generated `.medusa/server/package.json` (+ lockfile if present), so it stays cached across most builds.
</objective>

Key fact: the package.json consumed by `npm install --omit=dev` is GENERATED by `bunx medusa build` inside `.medusa/server`. It does NOT exist in source. Therefore we cannot copy it from the host before the build — we must split into multi-stage and cherry-pick the generated manifests as the cache key.

  <name>Task 1: Convert Dockerfile to multi-stage with cached prod-deps layer</name>
  <files>Dockerfile</files>
  <action>
Rewrite `/Dockerfile` as a two-stage build. Keep the existing builder logic byte-for-byte — only change is splitting into `builder` and `runtime` stages and reordering the runtime stage so `npm install --omit=dev` runs on a layer keyed only on the generated `.medusa/server/package*.json`.

**Required structure:**

```dockerfile
# ---------- Stage 1: builder ----------
FROM oven/bun:1-alpine AS builder

RUN apk add --no-cache python3 make g++ nodejs npm jq curl

WORKDIR /app

# Copy workspace manifests and lockfile for layer caching
COPY package.json bun.lock turbo.json ./
COPY apps/api/package.json ./apps/api/
COPY packages/cli/package.json ./packages/cli/
COPY packages/core/package.json ./packages/core/
COPY packages/types/package.json ./packages/types/
COPY packages/dashboard-sdk/package.json ./packages/dashboard-sdk/
COPY packages/dashboard-shared/package.json ./packages/dashboard-shared/
COPY packages/client/package.json ./packages/client/
COPY packages/registry/package.json ./packages/registry/
COPY packages/admin/package.json ./packages/admin/
COPY packages/vendor/package.json ./packages/vendor/
COPY packages/providers/payout-stripe-connect/package.json ./packages/providers/payout-stripe-connect/
COPY integration-tests/package.json ./integration-tests/

# Remove turbo (native binary that fails to extract in Alpine) before install
RUN jq 'del(.devDependencies.turbo) | del(.packageManager)' package.json > pkg.tmp && mv pkg.tmp package.json

RUN bun install

# Copy all source
COPY apps/api/ ./apps/api/
COPY packages/ ./packages/

# Build workspace packages in dependency order (types -> cli -> core)
RUN cd packages/types && bun run build
RUN cd packages/cli && bun run build
RUN cd packages/core && bun run build || true

# Build the Medusa API (generates /app/apps/api/.medusa/server/*)
RUN cd apps/api && bunx medusa build

# Remove bun-specific .npmrc inside the built server output before runtime stage picks it up
RUN rm -f /app/apps/api/.medusa/server/.npmrc || true

# ---------- Stage 2: runtime ----------
FROM oven/bun:1-alpine AS runtime

RUN apk add --no-cache nodejs npm

WORKDIR /app/apps/api/.medusa/server

# Cache key: generated server manifests ONLY. This layer stays cached unless
# the built package.json/package-lock.json changes (which only happens when
# upstream apps/api/package.json or its transitive deps change).
COPY --from=builder /app/apps/api/.medusa/server/package.json ./package.json
# Copy lockfile if Medusa's build emitted one; tolerate absence with a glob.
COPY --from=builder /app/apps/api/.medusa/server/package-lock.json* ./

RUN npm install --omit=dev --legacy-peer-deps

# Now copy the rest of the built server output (source-shaped, changes every build)
COPY --from=builder /app/apps/api/.medusa/server/ ./

EXPOSE 9000

ENV NODE_ENV=staging
CMD ["npx", "medusa", "start"]
```

**Implementation notes / what to avoid and why:**

- DO NOT try to copy `apps/api/package.json` (the source one) into the runtime stage and run npm install there. The runtime install must operate on the GENERATED `.medusa/server/package.json` produced by `bunx medusa build`; that file has the correct, pruned dep list Medusa expects. Using the source manifest will install the wrong tree.
- DO NOT collapse the two `COPY --from=builder` steps into one. The whole point is that the manifest copy + `npm install` must form a Docker layer that is cache-keyed only on the manifest contents, separate from the much-larger build-output copy.
- The lockfile copy uses a trailing `*` (`package-lock.json*`) so the build does not fail if Medusa's build did not emit one. If no lockfile exists, the resulting layer cache key still includes only `package.json`, which is fine.
- Keep `rm -f .npmrc` BEFORE the manifest copy in the builder stage so the runtime stage never inherits a bun-specific `.npmrc` that would confuse npm. (Original Dockerfile did `rm -f /app/apps/api/.npmrc`; we do the equivalent on `/app/apps/api/.medusa/server/.npmrc` because the runtime stage's WORKDIR is the server directory.)
- Runtime stage installs `nodejs npm` only — no `python3 make g++ jq curl`. Native build deps are needed only in the builder. This is a free image-size win, but the primary reason here is keeping the runtime stage minimal so its layers stay stable.
- The final `COPY --from=builder /app/apps/api/.medusa/server/ ./` copies on top of the existing `node_modules` from `npm install`. Docker COPY merges: source files overwrite, but `node_modules` stays because the builder stage doesn't have a `node_modules` inside `.medusa/server` (the builder never ran npm install there in the new layout). Verify this assumption (next task).
- `CMD` and `EXPOSE` and `ENV NODE_ENV=staging` MUST be preserved exactly — docker-compose and any health checks depend on them.

**One subtle correctness check, in the builder stage:** the original Dockerfile ran `RUN ls /app/apps/api/.medusa/` and `RUN ls /app/apps/api/.medusa/server/` for diagnostics. Drop these — they are noise and each adds a (tiny) cached layer. If you want to preserve diagnostic output, fine, but they are not required.
